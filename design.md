# MoShiRPC 设计文档（持续更新）

本文件用于同步记录 MoShiRPC 后续需要实现的模块、边界与接口契约，目标是让代码实现与论文描述能够保持一致，并为后续迭代提供可执行的工程路线图。为了便于你审查与二次编辑，本文档中所有需要你补充、确认或根据最终实现调整的内容，统一使用显眼标记：`【待你补充/修改】`。

## 1. 项目定位与目标

MoShiRPC 是一个面向 Linux 环境的轻量级 RPC 框架项目，主要关注在高并发长连接条件下的低延迟调用体验，以及在真实工程中可落地的稳定性与可维护性。框架的核心价值不只是“把请求发出去并拿到响应”，还应当覆盖一次调用从连接建立、报文解析、handler 执行、错误返回、超时回收，到日志追踪与测试验证的完整生命周期，从而让业务侧能够以接近本地函数调用的方式完成跨进程、跨主机协作。

该项目的最终形态应当满足两个层面的要求：其一是核心链路的性能与正确性，确保在粘包拆包、半包、短写、网络抖动与对端异常关闭等情况下仍能稳定工作；其二是工程化能力，至少提供可配置项、统一错误语义与可观测性基础，使框架能够被持续演进而不在规模扩大后失控。

## 2. 一次调用的最小模型（必须先定 API 与语义）

MoShiRPC 的最小可用模型建议以“三个入口”稳定下来：客户端 `RpcClient::Invoke`、服务端 `RpcServer::Register`，以及服务端生命周期 `Start/Stop`。在此模型中，客户端负责将一次调用封装为请求消息并发送到目标服务端，同时在本地建立 `request_id → promise/future` 的关联；服务端负责接收字节流并完成协议解析，在解析出完整请求后将其分发到已注册的 handler 执行，最后将响应写回同一连接并携带相同的 `request_id`，以便客户端完成匹配与唤醒等待者。

关于同步与异步语义，建议让 `Invoke` 同时支持阻塞等待与异步返回两种方式，默认以 future 的形式表达异步结果，并在上层通过 `.get()` 或回调等方式决定是否阻塞。超时语义应当被设计为一次调用的默认属性，既要支持客户端超时后不再等待，也要支持服务端在收到过期请求时尽早丢弃，避免将算力消耗在无意义的工作上。`【待你补充/修改】` 你需要最终确认：超时是仅在客户端生效，还是要作为 deadline 字段传播到服务端并参与服务端执行治理。

## 3. 模块划分与依赖关系（建议保持单向依赖）

整体上建议将框架拆分为网络通信层、协议与编解码层、调用语义层与工程化支撑层四个部分，并使依赖方向尽可能单向，避免后期演进时产生环形依赖。网络通信层负责连接与事件循环，协议层负责 framing 与消息编解码，调用语义层负责注册与分发、请求表与超时回收，工程化支撑层负责日志、配置、错误码、测试与基准工具。Python 绑定应被视为“上层接口适配层”，它不应反向影响底层模块的接口稳定性。

## 4. 网络通信层（EventLoop/Channel/Connection/TcpSocket）

当前工程已具备基于 epoll 的 `EventLoop` 与 `Channel` 抽象，以及基础的 `TcpSocket` 封装，但要支撑 RPC 级的并发与可靠性，还需要补齐面向连接的完整抽象与写路径工程化。

Connection 抽象应当成为网络层的中心对象，它需要持有输入缓冲与输出缓冲，并维护连接状态机，使得读路径能够将数据持续写入输入缓冲、触发协议层的流式解析，而写路径能够将待发送的数据积累在输出缓冲中并通过 `EPOLLOUT` 驱动逐步 flush，从而正确处理短写与慢客户端问题。为了实现线程模型清晰，Connection 还应当具备明确的 thread-affinity，即连接上的所有 I/O 操作与状态变更默认在其所属的 EventLoop 线程内完成；跨线程投递任务时，需要通过唤醒机制（例如 eventfd）将任务切回 I/O 线程执行，避免锁把状态撕裂。

在 socket 工程化方面，需要系统性完成非阻塞、`accept4`、`SO_REUSEADDR`、`TCP_NODELAY`、keepalive 等选项设置，并对读写返回值为 0、`EINTR`、`EAGAIN`、`EPOLLHUP/EPOLLRDHUP/EPOLLERR` 等边界进行统一处理。`【待你补充/修改】` 你需要确认是否要支持多 EventLoop（多 Reactor）模型，以及连接在多个 EventLoop 之间的分配策略是轮询、哈希还是基于负载反馈。

## 5. 协议帧（Framing）与解析器（Parser）

RPC 运行在 TCP 字节流之上，粘包、拆包与半包是必须显式处理的问题，因此需要在协议中引入稳定的 framing 机制。建议的协议帧由固定长度头部与变长 body 组成，其中头部至少包含 magic、version、type、flags、request_id、timeout/deadline、body_len 等字段，以便解析器能够在输入缓冲区中进行增量解析，并在任意切片输入的情况下持续推进状态机。

解析器的实现建议采用“可流式消费”的状态机模型，它从输入缓冲区读取可用字节，并在头部未齐时继续等待，在头部齐全后根据 `body_len` 判断是否已收齐 body。为了安全性与稳定性，解析器应提供 `max_frame_size` 等硬阈值，在发现异常长度、未知版本或 magic 不匹配时立即返回协议错误，并触发连接关闭或降级处理，避免内存被恶意输入拖垮。`【待你补充/修改】` 你需要确定 checksum、压缩与加密是否进入 V1 协议，如果进入则必须把协商方式与失败路径写进协议语义与实现细节。

## 6. 编解码层（Codec）与消息语义（Envelope）

在 framing 之上，需要定义一层稳定的“调用语义消息”，以便让不同语言与不同业务都能通过统一 envelope 结构表达一次 RPC 调用。建议 envelope 至少包含 service、method、headers、status 与 payload，其中 payload 是序列化后的参数或返回值，headers 用于承载扩展字段（例如 trace_id、deadline、客户端版本等），status 用于表达业务错误与框架错误的统一返回格式。

Codec 的接口应当允许多种序列化实现并可插拔扩展。为了降低早期调试成本，可以先支持 JSON；为了提升性能与类型约束，可以在后续引入 Protobuf 等二进制协议。无论选择哪种序列化方式，都需要明确定义跨语言的类型边界，例如整数范围、字符串编码、可选字段与默认值语义，并在解码阶段加入输入校验与反序列化防御，避免恶意 payload 触发异常路径。`【待你补充/修改】` 你需要最终确定 V1 支持的序列化集合，以及服务端对未知序列化类型的处理策略（拒绝、降级或兼容）。

## 7. 服务注册、分发与执行模型（Dispatcher/ServiceRegistry）

服务端需要提供可维护的注册机制，使得业务侧能够以较少样板代码将函数注册为可远程调用的 handler。ServiceRegistry 的核心职责是维护 `service + method → handler` 的映射，Dispatcher 的核心职责是在请求到达时执行路由、参数校验、异常捕获与错误码映射，并将 handler 投递到线程池执行，从而保证 I/O 线程不被耗时逻辑阻塞。

为保证框架的稳定性，handler 的执行应当被包裹在统一的保护边界中，业务异常应被捕获并转换为标准化的错误返回，而不是让异常跨越线程边界导致进程崩溃。对于执行时间过长的 handler，框架应当明确治理策略，例如仅记录告警、仅在客户端侧超时返回，或在服务端侧引入可取消的执行机制。`【待你补充/修改】` 你需要确认是否真的要“自动中断”长耗时 handler，因为在 C++ 中强制中断线程通常不安全，实际可行方案往往是协作式取消或隔离执行环境。

## 8. 客户端（RpcClient）与请求复用（Multiplexing）

客户端的关键能力是同一连接上的并发调用复用，它需要维护 `request_id → promise/future` 的请求表，并在响应到达时完成匹配与唤醒。为了避免请求表泄漏，需要将超时回收作为硬性机制，并在连接断开时对所有 in-flight 请求返回明确错误。客户端还应当支持连接池或至少支持单连接复用，并对重连与退避策略给出明确的设计边界。

对于重试与幂等，需要先在语义上划清边界，明确哪些错误可以重试、哪些错误不应重试，避免框架在业务不知情的情况下导致副作用被重复执行。`【待你补充/修改】` 你需要决定 V1 是否实现重试，如果实现则必须要求服务端方法具备幂等声明或由调用方显式开启。

## 9. 服务端（RpcServer）生命周期与连接治理

服务端侧需要提供清晰的启动与停止语义，保证 Stop 返回后不再接收新连接，并能够优雅关闭已有连接。连接治理应至少覆盖心跳、空闲断开与 half-close 处理，并在异常事件（ERR/HUP/RDHUP）下完成资源释放与请求回收。对于慢客户端或恶意客户端，需要在输出缓冲高水位、单连接并发上限与 payload 大小上限等方面实现保护，避免拖垮整个进程。

## 10. 服务发现与负载均衡（可分阶段落地）

服务发现与负载均衡可以按演进路线分阶段完成。第一阶段支持静态地址列表，提供轮询或随机策略即可满足开发与实验；第二阶段引入注册中心（如 etcd/consul/zookeeper）并实现注册、续租、下线与 watch，同步维护可用实例集合；第三阶段加入健康检查、熔断与故障转移，在服务抖动时保持整体可用性。`【待你补充/修改】` 你需要选择注册中心的落地方案，以及在论文与实现中保持一致。

## 11. 可观测性、日志与错误体系

框架需要统一的错误体系，明确区分网络错误、协议错误、超时错误与业务错误，并保证错误在客户端与服务端都能以可序列化的形式传递。日志体系应当为 RPC 关键路径提供稳定字段，例如 request_id、service、method、remote_addr、latency，并在连接事件、解析失败、handler 异常与超时回收等路径留下足够信息。随着项目演进，可以逐步引入 metrics 与分布式追踪，但在 V1 阶段至少应保证日志可用于定位大部分问题。

## 12. Python 绑定（pybind11）与示例工程

如果项目目标包含跨语言互操作，则 Python 绑定应当提供最小可用的 client/server API，并给出一个可运行的 demo 服务，用于验证序列化、超时与并发复用等能力。绑定层应尽量薄，只做类型转换与生命周期托管，不要在绑定层复制底层逻辑，避免出现“C++ 一套语义，Python 一套语义”的分裂。`【待你补充/修改】` 你需要确认 Python 是否为必须交付项，以及绑定层是否需要支持异步（例如 asyncio）风格接口。

## 13. 测试、基准与验收标准

测试需要覆盖两个层面：其一是基础组件的单元测试，例如 EventLoop 的 Stop/唤醒语义、定时器的并发与取消语义、线程池的提交与销毁语义、缓冲区的读写边界；其二是 RPC 端到端集成测试，覆盖单次调用、并发调用、同连接复用、超时回收、服务端异常、半包输入与乱序写等真实场景。基准测试需要至少提供吞吐与延迟分位数据，并在不同 payload 大小、不同并发度、不同序列化方式下给出趋势结论，从而使论文中的“性能提升”有可复现的证据链。
